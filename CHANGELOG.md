## [1.1.1] - 2026-06-28

### Added
- **Patent Grant (Section IV)**
  - Non-exclusive, worldwide, royalty-free, irrevocable patent license from contributors
  - Applies to patent claims necessarily infringed by contributions
  - Automatic termination if user initiates patent litigation

- **Internal Use Exception (Exception 2)**
  - No renaming required when modifying software for internal use only
  - Applies regardless of whether used as library or standalone application
  - Clarifies that internal modifications are exempt from renaming obligation

- **Unmodified Distribution Exception (Exception 3)**
  - No renaming required when distributing verbatim copies
  - Must retain all copyright notices and permission notices

- **License Versioning (Section X)**
  - SPDX-License-Identifier: `Dryfish-1.1.1`
  - Users may use any version of the license
  - No forced upgrade to future versions

- **Compatibility Section (Section VIII)**
  - Lists compatible licenses: MIT, BSD, Apache-2.0
  - Lists incompatible licenses: GPL-2.0, GPL-3.0, AGPL-3.0

- **Governing Law (Section IX)**
  - Specifies which laws govern the license
  - Adds legal enforceability

- **Expanded "PROJECT NAME" Definition**
  - 8 specific categories listed:
    1. Repository name
    2. Package/Namespace name
    3. Binary/Executable name
    4. Display name
    5. Logo and branding elements
    6. Marketing terms
    7. Domain names and URLs
    8. Social media handles

- **Violation Examples (Section III)**
  - 6 types of violations with concrete examples:
    - Only case change
    - Only punctuation added
    - Only numbers added
    - Only spacing changed
    - Abbreviations
    - Prefixes/suffixes without core change

- **Compliance Examples (Section III)**
  - 3 types of compliance with concrete examples:
    - Different root word
    - Completely different name
    - Different branding concept

- **SPDX Identifier**
  - `Dryfish-1.1.1` added to license header

- **Repository URL**
  - Added at the end of license: https://github.com/dryfish09/DryFishLicense

- **Important Reminders Section**
  - 4 reminders about renaming obligations and exceptions
  - Contact information for clarification

### Changed
- **Section III (Project Renaming Obligation)**
  - Significantly expanded with integrated examples
  - Added 3 exceptions (Library, Internal Use, Unmodified Distribution)
  - Clarified "Clearly different" criteria
  - Added 8 specific categories of "PROJECT NAME"

- **License Header**
  - Added SPDX-License-Identifier
  - Updated version to 1.1.1

### Fixed
- **Ambiguity in "PROJECT NAME" definition**
  - Previously vague, now explicitly defined with 8 categories
- **Missing Patent Grant**
  - Added to protect users from patent litigation
- **Missing Internal Use Exception**
  - Previously unclear whether internal modifications require renaming
- **Missing Unmodified Distribution Exception**
  - Previously unclear whether verbatim copies require renaming
- **Missing License Versioning**
  - Previously didn't specify versioning rules

## [1.1] - 2026-06-27

### Added
- **Library Exception Clause (Exception 1)**
  - No renaming required when using as library/dependency
  - Original package name may remain unchanged
  - Applies to both modified and unmodified library usage

- **Concrete Usage Examples**
  - Using as a library (no renaming)
  - Modifying a library for internal use (no renaming)
  - Forking standalone application (must rename)
  - Forking and distributing as separate library (must rename)

### Fixed
- **Library/Dependency Issue from v1.0**
  - Previously required renaming of package/class names
  - Now explicitly exempts library usage

## [1.0] - 2026-06-26

### Added
- **Initial Release**
  - Complete license text
  - Grant of Rights (Section I)
  - Attribution Requirement (Section II)
  - Project Renaming Obligation (Section III) - Mandatory
  - Disclaimer of Warranty (Section IV)
  - Termination Clause (Section V)
  - Acceptance Clause (Section VI)

### Features
- ✅ MIT-style permissions (use, modify, distribute)
- ✅ Commercial use allowed
- ✅ Proprietary integration allowed
- ✅ Mandatory renaming for standalone forks
- ✅ Attribution requirement
- ✅ Automatic termination for violations

## 🔄 Migration Guide

### Upgrading from v1.0 to v1.1

**Changes:**
- Library usage no longer requires renaming
- Package/class names can stay the same

**Action:**
- ✅ No action needed if using as library
- ✅ Continue to rename standalone forks

---

### Upgrading from v1.1 to v1.1.1

**Changes:**
- Patent Grant added (beneficial to you)
- Internal Use Exception added (beneficial to you)
- Unmodified Distribution Exception added (beneficial to you)
- Expanded definition of "PROJECT NAME"

**Action:**
- ✅ No action needed for existing users
- ✅ Benefits apply automatically
- ✅ Review new "PROJECT NAME" definition to ensure compliance
- ✅ Review Patent Grant terms
